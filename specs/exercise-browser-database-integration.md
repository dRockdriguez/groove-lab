# Spec: Exercise Browser Database Integration

**Status:** Implemented
**Version:** 0.1.0
**Last updated:** 2026-03-16

---

## Problem

Currently, the exercise browser in the homepage displays mock exercises from a static data file
(`mockExercises.ts`). Real exercises are being uploaded to the database via the import feature,
but they are not visible to users browsing the app.

Users cannot discover or practice with the exercises they have uploaded. The homepage needs to
fetch real exercises from the database and organize them by instrument and section, replacing
the mock data entirely.

---

## User Stories

1. **As a musician**, I want to see my uploaded exercises in the exercise browser when I visit
   the homepage so that I can immediately find and practice with the content I've imported.

2. **As the system**, I want to fetch exercises from the database organized by instrument and
   section so that the UI receives data in the correct format without additional transformation.

3. **As a musician**, I want the exercise list to update without manual page reload when new
   exercises are imported so that my workflow is seamless.

4. **As a developer**, I want the database queries to be efficient and return exercises grouped
   by instrument and section for consistent UI rendering.

---

## Acceptance Criteria

### Backend: GET /exercises Endpoint

- [x] A new endpoint `GET /exercises` returns all exercises grouped by instrument.
- [x] Response format matches the frontend's `InstrumentExercises[]` type:
  ```ts
  {
    instrumentType: 'electronic-drums' | 'bass-guitar' | 'guitar',
    sections: [
      {
        id: string,          // derived from folder name (e.g., "basic-rhythms")
        title: string,       // human-readable section title (e.g., "Basic Rhythms")
        exercises: [
          {
            id: string,      // exercise ID (e.g., "drums-basic-1")
            title: string,   // exercise title (e.g., "Exercise 1")
            description: string  // exercise description
          }
        ]
      }
    ]
  }[]
  ```
- [x] The endpoint groups exercises by `instrument` field from the database.
- [x] Within each instrument, exercises are grouped by the `section` folder name.
- [x] If an instrument has no exercises, its entry is omitted from the response (optional: included with empty sections array).
- [x] The endpoint returns HTTP 200 with the exercises array, or HTTP 500 on database error.
- [x] Response time is under 200ms for typical datasets (< 100 exercises).

### Exercise Sections and Metadata

- [x] Sections are derived from the folder structure: instruments are top-level, folders within instruments are sections.
- [x] Section `id` is the folder name (e.g., "basic-rhythms", "line-1", "chords-1").
- [x] Section `title` is generated by humanizing the folder name (e.g., "Basic Rhythms", "Line 1", "Chords 1").
- [x] If section title humanization is unclear, store a metadata field in the database for human-readable titles (out of scope for now; generate from folder name).

### Frontend: Homepage Data Fetching

- [x] The homepage fetches exercises from `GET /exercises` on mount via `useEffect`.
- [x] Data is fetched in the browser (client-side) via `useEffect` + `useState`.
- [x] While fetching, the page displays a loading state (spinner or placeholder).
- [x] If the fetch fails (network error, server error), the page displays a user-friendly error message and a retry button.
- [x] On successful fetch, the `ExerciseBrowser` component receives the real exercises array.
- [x] The `ExerciseBrowser` renders exercises in the same manner as with mock data (no UI changes needed).
- [x] If the API returns an empty array, the exercise browser shows "No hay ejercicios disponibles" for all instruments.

### Integration & Backward Compatibility

- [x] Existing types in `@groovelab/types` (`InstrumentExercises`, `ExerciseSection`, `Exercise`) remain unchanged.
- [x] No changes needed to `ExerciseBrowser`, `ExerciseCard`, `ExerciseSectionList`, or `InstrumentButton` components.
- [x] The mock data file (`mockExercises.ts`) is retained (required by existing tests that import it directly; production code no longer imports it).
- [x] Navigation links to exercises still use the format `/practice/{instrumentType}/{exerciseId}`.

### Error Handling

- [x] Network errors (fetch fails) display a user-facing message: "No pudimos cargar los ejercicios. Intenta de nuevo."
- [x] Server errors (5xx) display the same message.
- [x] Failed requests can be retried without page reload.
- [x] The page remains usable even if the exercises endpoint is temporarily unavailable.

## Technical Notes

### Backend Implementation

**Files to create or modify:**
- `apps/api/app/routes/exercises.py` — new file with GET /exercises endpoint
- `apps/api/app/services/exercise_service.py` — optional service layer for database queries
- `apps/api/app/main.py` — register the exercises router

**Database Query Strategy:**
- Query all exercises from the `exercises` table (assume schema from `exercise-import-backend.md`).
- Group by `instrument` and `section` in Python (not SQL group-by, for simplicity and flexibility).
- Transform to `InstrumentExercises[]` response format.

**Assumptions:**
- Exercises table exists with columns: `id`, `title`, `description`, `instrument`, `section`, `folder_name`.
- MIDI/MP3 files are stored on disk; only metadata is queried here.

### Frontend Implementation

**Files to create or modify:**
- `apps/web/src/components/Homepage.tsx` — add `useEffect` to fetch `/api/exercises`
- `apps/web/src/pages/index.astro` — pass fetched exercises to `Homepage` (if server-side fetch) OR pass loader function (if client-side)
- Remove or comment out `mockExercises` import

**Data Fetching Approach:**
- Option A: Client-side fetch in `Homepage.tsx` using `useEffect` + `useState`
  - Pros: Standard React pattern, data freshness
  - Cons: Loading state visible, potential layout shift
- Option B: Server-side fetch in `index.astro` using Astro loader
  - Pros: Data ready before render, better UX
  - Cons: Adds Astro-specific logic, requires SSR

**Recommendation:** Option B (server-side via Astro) if feasible; Option A if simpler to implement.

### Error Handling Implementation

**Loading state:** Skeleton loader or spinner component (optional; can defer to v0.2).
**Error state:** Inline error message with retry button.
**Retry logic:** `handleRetry` function that re-fetches data.

## Out of Scope

- Pagination, infinite scroll, or lazy-loading of exercises
- Filtering or searching by exercise name, section, or difficulty
- Exercise metadata beyond `title`, `description` (e.g., difficulty, duration, BPM)
- Caching strategies or stale-while-revalidate patterns
- Humanizing section titles via database metadata (generate from folder names for now)
- Real-time updates when new exercises are imported (manual page refresh required)
- Analytics or usage tracking
